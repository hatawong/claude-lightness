# 指挥官给研究员写任务的指南

> 读者: **指挥官 (commander)** — 在多 Agent 协作中给研究员 (researcher) 派 task 的角色
> 前置: 先读 [04-multi-agent.md](04-multi-agent.md) (多 Agent 协作框架) + [agents/researcher/ACTOR.md](../../agents/researcher/ACTOR.md) (研究员本身的角色定义)

---

## 用途

任务 prompt 用 **role / why / what / how** 四字段 (+ 可选 `deps / optDeps`) 描述, 由指挥官写给研究员执行。本文档约定 **研究员角色** 这些字段的填写规范。

**注**: 字段载体 (JSON / Markdown / 别的) 由项目定, 本文档只约束**字段内容**怎么写。

---

## 研究员是谁

研究员负责**探索和分析**——调研技术方案、分析源码架构、评估产品功能、研究用户体验等，输出结构化的分析报告。产出是 report 文件。

研究员**不做实现**——发现和建议写在 report 里，由指挥官或调度者决定是否采纳。研究员不写代码、不改配置、不做决策。

---

## WHY 怎么写

> 任务 prompt 的 WHY 回答：**为什么需要这项研究**。

规则：
1. **写因果链，不压缩**——从背景到当前认知状态到为什么需要研究，每一步都写清楚
2. **不相关的历史主动排除**——显式写「你不需要了解 X，按需查阅弱依赖即可」
3. **站在新人的视角**——假设 Actor 对这个任务的背景一无所知

重点：说清楚**为什么要研究这个东西**——我们想从中学到什么？当前已经知道什么、缺少什么？

示例（好）：
```
某开源项目（314K stars，TypeScript）是当前最大规模的开源 AI 助手平台。我们正在设计多 Agent 编排系统，
需要对标该项目的架构来验证和改进我们的设计。

当前了解停留在博客和调研报告层面（记忆架构、泳道并发概念），缺少源码级的架构理解。
```

示例（差）：
```
研究某项目源码。
```

## WHAT 怎么写

> 任务 prompt 的 WHAT 回答：**研究什么、交付什么结构的报告**。

**要做什么**：说清楚研究范围和方向。第一轮通常是开放式探索（"不带预设框架，自由探索"），后续轮次可以聚焦特定模块。

**当前状态**（可选但推荐）：列出已有的研究成果和已知信息，避免研究员重复工作。

**交付物**：编号列表，每条是报告的一个章节。研究员的交付物通常是分析报告的结构：

```
交付物：
1. 项目结构总览（核心目录和文件的用途）
2. 架构分层（从入口到执行的调用关系）
3. 关键模块识别（哪些模块和我们关注的问题相关）
4. 初步发现（源码中看到的有价值的设计模式）
5. 建议下一步深入的方向
6. 如果遇到无法解决的问题，在报告中标注为 raise 项
```

**注意**：模板会自动追加一行 `产出写入 {reportPath}`，不需要在交付物里重复写路径。

示例（好）：
```
读源码，输出一份基础架构分析报告。

这是第一轮探索——不带预设框架，自由探索源码结构，建立全景图。后续轮次再决定深入哪里。

交付物：
1. 项目结构总览
2. 架构分层
3. 关键模块识别
4. 初步发现
5. 建议下一步深入的方向
```

示例（差）：
```
分析源码。
```

## HOW 怎么写

> 任务 prompt 的 HOW 回答：**怎么研究**。通常包含流程、规范、raise 机制、report 格式。

**流程**：告诉研究员按什么顺序工作。典型步骤：
1. 获取源码到本地（git clone / git pull）
2. 从 README 和 package.json 入手，理解项目定位
3. 浏览目录结构，识别核心模块
4. 读关键入口文件，追踪调用链
5. 自己决定深入的优先级——不需要覆盖所有模块
6. 写报告

**规范**：告诉研究员哪些约束必须遵守。典型规范：
- 源码引用要精确：`文件路径:行号` 或 `文件路径:函数名`
- 不确定的标注「需要进一步确认」
- 不要猜测——看到什么写什么
- 所有源码分析基于本地文件，不通过 GitHub API

**raise 机制**：告诉研究员遇到无法解决的问题时怎么办——在 report 中标注为 raise 项，每项包含三要素：发现了什么（事实）、为什么处理不了（判断）、建议方向（建议，不是决策）。典型 raise 场景：源码是 minified 无法分析、仓库过大无法在时间内覆盖、关键模块需要运行时调试才能理解。

**report 格式**：告诉研究员交付的 report 应该长什么样。研究员 report 通常包含：按交付物列表逐章分析、每章有具体的源码引用、raise 项（如有）。

**注意**：新 session 的第一个任务必须写 raise 机制和 report 格式。同 session 续跑的后续任务可以省略。

示例（好，新 session 第一个任务）：
```
### 流程

1. 克隆仓库到本地（如果已存在则 git pull）
2. 所有源码分析基于本地文件，不通过 GitHub API
3. 从 README 和 package.json 入手
4. 浏览 src/ 目录结构，识别核心模块
5. 读关键入口文件，追踪调用链
6. 自己决定深入的优先级——不需要覆盖所有模块
7. 写报告

### 规范

- 源码引用要精确：文件路径:行号 或 文件路径:函数名
- 不确定的标注「需要进一步确认」
- 不要猜测——看到什么写什么

### raise 机制

遇到无法解决的问题，在 report 中标注为 raise 项，包含：
- 发现了什么（事实）
- 为什么当前任务处理不了（判断）
- 建议的方向（建议，不是决策）

### report 格式

report 按交付物列表的章节结构写，每章有具体的源码引用。
```

示例（差）：
```
读源码写报告。
```

## DEPS / OPTDEPS 怎么选
> 任务的强依赖 / 任务的弱依赖

**deps（强依赖）**：不读就做不了研究。选择标准——如果 Actor 跳过这个文档，会在哪一步无法继续？

**optDeps（弱依赖）**：按需查阅。选择标准——跳过也能完成研究，但某些场景下需要参考。

研究员典型依赖：

| 类型 | 文档 | 理由/场景 |
|------|------|----------|
| 强 | 项目工作模式 | 理解我们为什么要研究这个项目 |
| 强 | 设计文档 | 理解当前架构，知道研究成果要对接什么 |
| 强 | 前置研究报告（如有） | 前一轮研究员的产出，避免重复研究、在已有认知上继续 |
| 弱 | 历史研究报告 | 更早的研究记录，按需查阅了解演进过程 |

示例：
```json
"deps": [
  { "path": "work-patterns.md", "desc": "项目工作模式", "why": "理解我们为什么要研究这个项目" },
  { "path": "design.md", "desc": "设计文档", "why": "理解当前架构，知道研究成果要对接什么" },
  { "path": "研究员-5-report.md", "desc": "前一轮研究报告", "why": "在已有认知上继续，避免重复" }
],
"optDeps": [
  { "path": "研究员-1-report.md", "desc": "第一轮研究报告", "when": "需要了解研究演进过程时" }
]
```
